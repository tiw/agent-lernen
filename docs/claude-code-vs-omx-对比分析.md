# Claude Code vs oh-my-codex 对比分析

## 为什么能力完备的 Agent 还需要外挂层？

---

## 一、定位差异

| 维度 | Claude Code | oh-my-codex (OMX) |
|------|-------------|-------------------|
| **定位** | 终端原生 Agent 操作系统（底层运行时） | 编排增强层（工作流调度器） |
| **类比** | 操作系统内核 | 桌面环境 + 任务管理器 |
| **目标** | 提供完整的 Agent 能力 | 让 Agent 按流程有序工作 |
| **原生能力** | 40+ 工具、多 Agent、记忆、权限治理 | 依赖底层 CLI（Codex），专注编排 |

> OMX 是给 OpenAI Codex CLI 的，但这里的核心问题是：**为什么一个能力完备的 Agent 还需要外挂层？**

---

## 二、Claude Code 已有的长时间自主编程能力

从架构分析看，Claude Code 已经非常强大：

1. **KAIROS 实时感知** — 每 15 秒自动唤醒，自主探索和行动
2. **AutoDream 记忆巩固** — 空闲时自动修剪、整合记忆
3. **多 Agent 协作** — Team/Coordinator 模式，支持并行调度
4. **持久化记忆** — Session/Project/Team 三级记忆体系
5. **上下文压缩** — Token 超限时自动摘要
6. **会话恢复** — 完整的序列化/反序列化
7. **Token 预算管理** — 每 Turn 可配置最大消耗
8. **Git 工作树隔离** — 独立分支工作，不影响主工作区

从底层能力看，Claude Code 已经具备了长时间自主运行的基础设施。

---

## 三、为什么还需要 OMX 这样的外挂层？

核心原因：**能力不等于秩序**。Claude Code 有强大的原子能力，但缺少**结构化的工程化工作流编排**。

### 1. 工作流固化 vs 临场发挥

| | Claude Code 原生 | + OMX 类外挂 |
|---|---|---|
| **工作流** | Agent 自主决定下一步做什么 | 预定义 SOP：`plan → exec → verify → fix` |
| **角色分工** | 需要手动配置 Team | 32 个预定义角色，开箱即用 |
| **流程控制** | 依赖模型推理决定 | 通过关键词显式触发（`$plan`、`$architect`） |

Claude Code 的 Agent 很聪明，但"聪明"在长时间任务中是个双刃剑。OMX 通过固化流程，确保：
- 需求不清时先追问（`$deep-interview`），而不是自信开编
- 动手之前先规划（`$plan`），而不是直接写代码
- 复杂任务先架构分析（`$architect`），而不是边做边改

**这不是能力问题，是工程纪律问题。**

### 2. 状态持久化的深度差异

| | Claude Code | OMX |
|---|---|---|
| **记忆内容** | 会话摘要、项目知识、用户偏好 | 计划、执行日志、运行状态、角色分配 |
| **持久化位置** | `.claude/memory.md` | `.omx/` 完整目录结构 |
| **恢复能力** | 恢复对话上下文 | 恢复整个工作流程进度 |

Claude Code 的记忆偏向"知识记忆"（记住了什么），OMX 的记忆偏向"过程记忆"（干到什么程度了、谁在干什么、下一步该谁）。长时间任务中，**过程记忆比知识记忆更重要**。

### 3. 异步作业与超时突破

Claude Code 的 QueryEngine 是同步循环：采样 → 工具执行 → 结果注入 → 再采样。如果单个工具调用超时，虽然有重试机制（最多 3 次），但整体仍受限于单次 API 调用的超时。

OMX 的异步 job 系统：
- 长时任务提交后后台运行，不阻塞主循环
- 基于 tmux 的终端会话天然支持长时间运行
- 可以随时查看进度、恢复、或终止

### 4. 自动模型路由 vs 固定模型

Claude Code 在 Team 模式下可以指定不同 Agent 用不同模型（如 Architect 用 opus，Coder 用 sonnet），但需要手动配置。

OMX 的自动模型路由：
- 根据任务复杂度自动选择模型
- 简单任务用小模型省 token，复杂任务用大模型保质量
- 推理力度（reasoning effort）也自动映射

长时间自主编程中，这种自动路由能显著降低成本同时保证关键环节的推理深度。

### 5. 并行执行的隔离策略

两者都支持并行，但 OMX 的工程化隔离更激进：
- 每个 Worker 运行在独立的 `git worktree` 中
- 天然防止文件冲突
- 最后有序合并

Claude Code 的 Team 模式也有隔离，但 OMX 的隔离更偏向"每个 Agent 是一个独立开发者"的工程范式。

---

## 四、本质总结

用一个比喻：

> **Claude Code 是一个全能的超级工程师** — 会写代码、会测试、会审查、会记住项目历史、会自主行动。
>
> **OMX 是一个项目经理 + 流程系统** — 它不写代码，但它确保这个超级工程师：先理解需求、再做方案、再动手、做完验证、遇到问题有人接手、长时间工作有进度记录。

| 维度 | 解决的问题 |
|------|-----------|
| Claude Code | **Agent 能力**（能做什么） |
| OMX | **Agent 秩序**（按什么顺序、什么节奏、什么规范去做） |

**能力强的 Agent + 好的工作流 = 可信赖的长时间自主编程**

缺少任何一环，长时间任务都会出问题：
- 只有能力没有秩序 → 做偏了、做乱了、做着做着忘了目标
- 只有秩序没有能力 → 流程再完美也干不出好代码

所以 OMX 这类外挂层的存在，不是因为 Claude Code 能力不够，而是因为**工程化工作流和运行时秩序是另一个维度的问题**，值得用专门的层来解决。
