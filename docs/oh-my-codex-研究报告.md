# oh-my-codex 研究报告：如何让长时间自主编程成为可能

## 核心定位

oh-my-codex（简称 OMX）的本质不是替代 Codex CLI，而是给它装了一层**"外挂大脑"和"稳定工作运行时"**。它解决的核心问题是：原生 AI 编程工具在长时间、复杂任务中会出现上下文丢失、执行无序、状态不连贯、单兵效率瓶颈等问题。

## 六大关键机制

### 1. 状态持久化 — 记忆不断线

所有计划、日志、记忆和运行状态自动写入 `.omx/` 目录。这意味着：
- 从"临时会话记忆"升级为"项目档案系统"
- 跨轮次对话不会丢失上下文（普通对话式 AI 最大的问题就是上一轮像 CTO，下一轮像刚入职第一天）
- 长期任务中断后可以恢复

### 2. 多智能体编排 — 从单兵到团队

OMX 内置了 **32 个专业化 AI 角色**：

| 类别 | 角色 |
|------|------|
| **编排层** | PM（任务分配）、Coordinator（并行管理）、Executor（执行） |
| **开发** | Frontend、Backend、Fullstack、Mobile、DevOps |
| **质量** | Tester、Reviewer、Security、Debugger |
| **专项** | Architect、Researcher、Data/ML、Writer |

团队模式基于 `tmux`/`psmux` 实现多 Worker 并行：
- Worker 运行在隔离的 `git worktree` 中，防止文件冲突
- 默认走结构化流水线：`plan → exec → verify → fix`
- 不仅解决多代理并行，还解决并行后的合并与冲突管理

### 3. 固化工作流 — 先想清楚再动手

通过 **Skills（技能）系统** 将常见开发动作抽象为可复用的 SOP：

| 指令 | 作用 |
|------|------|
| `$deep-interview` | 需求模糊时先追问澄清，避免"自信开编" |
| `$architect` | 分析系统边界、依赖、风险与权衡 |
| `$plan` | **强制先规划再动手**，减少返工 |
| `$executor` | 方向明确后专注执行，停止发散 |
| `$ralph` | 多步骤连续任务，高耐心顺序执行 |
| `$team` | 大型任务，多 Worker 并行协作后整合 |

### 4. 自动模型路由 — 资源最优分配

OMX 根据任务复杂度自动选择不同模型：

| 复杂度 | 模型 |
|--------|------|
| 实时 | gpt-5.3-codex-spark |
| 简单 | gpt-5-codex-mini |
| 标准 | gpt-5.2-codex |
| 复杂 | gpt-5.3-codex |
| 长时任务 | gpt-5.1-codex-max |

推理力度也自动映射：简单任务用 `none`，复杂调试用 `xhigh`。这保证了长时间运行时不会浪费 token，同时在关键环节投入足够算力。

### 5. 异步作业系统 — 突破超时限制

OMX 通过异步 job 系统解决工具调用超时问题。长时任务（如大型重构、全链路测试）可以可靠地完成，不会因为单次 API 调用超时而中断。

### 6. 实时监控面板 — `omx hud --watch`

长时间自主编程需要一个可视化监控界面，HUD 可以实时查看长期任务状态和多 Worker 进度，让开发者知道 AI 在干什么、干到哪了。

---

## 核心结论

> **AI 编程的瓶颈很多时候不是模型能力，而是缺少一个稳定的工作运行时。**

oh-my-codex 补上了这个运行时，包含四个要素：
- **秩序** — 角色分工、流程 SOP
- **流程** — plan → exec → verify → fix 的闭环
- **记忆** — 状态持久化到 `.omx/`
- **协作** — 多代理并行 + 隔离 + 有序合并

它不是"一键无 Bug"的神器，而是一个务实的基础设施层，让 AI 从"高效但随缘的自由职业者"变成"有项目经理、有架构师、有工作日志、有团队协作能力的系统化开发引擎"。
