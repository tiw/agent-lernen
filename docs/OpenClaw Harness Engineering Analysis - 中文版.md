## 控制论映射

### 1. 反馈系统

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **子代理注册表** | `agents/subagent-registry.ts` | 集中跟踪生成的子代理运行 |
| **父流中继** | `agents/acp-spawn-parent-stream.ts` | 从子到父的实时输出流 |
| **会话生命周期事件** | `agents/subagent-spawn.ts` | `emitSessionLifecycleEvent` 用于会话状态更改 |
| **Cron运行遥测** | `cron/types.ts` - `CronRunTelemetry` | 每次cron运行的令牌使用跟踪 |
| **会话存储** | `config/sessions.ts` | 基于JSON的会话持久化 |
| **子代理控制** | `agents/subagent-control.ts` | 用于编排的列表/终止/引导操作 |

**关键洞察**: 子代理注册表实现了 **集中式反馈** - 所有子代理运行都被跟踪，带有请求者来源，允许父会话监控、控制和接收完成通知。

### 2. 控制机制

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **深度限制** | `agents/subagent-spawn.ts` | `DEFAULT_SUBAGENT_MAX_SPAWN_DEPTH = 1`（可配置） |
| **每个代理的最大子代理数** | `agents/subagent-spawn.ts` | 默认每个会话5个活动子代理 |
| **最大并发性代理** | `config/agent-limits.ts` | 默认8个并发子代理 |
| **运行超时** | `agents/subagent-spawn-plan.ts` | 每个子代理的可配置超时 |
| **沙箱继承** | `agents/subagent-spawn.ts` | 沙箱会话不能生成非沙箱子代理 |
| **代理白名单** | `agents/subagent-spawn.ts` | `allowAgents` 配置控制跨代理生成 |
| **工具上下文限制** | `agents/subagent-capabilities.ts` | 基于深度的能力系统 |

**关键洞察**: 多层次控制，具有 **分层深度跟踪**（`getSubagentDepthFromSessionStore`）和基于委托深度的 **能力衰减**。

### 3. 通信基础设施

| 实现 | 位置 | 描述 |

| **Cron交付** | `cron/delivery.ts` | 计划作业的多平台交付 |

**关键洞察**: **以网关为中心的架构** - 所有代理通信都通过中央网关，该网关处理跨平台的路由、交付和会话管理。

### 4. 熵管理

| **附件清理** | `agents/subagent-attachments.ts` | 自动临时目录清理 |

## 可扩展性维度评估

### 时间可扩展性: ⭐⭐⭐⭐ (强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **持久会话** | 线程绑定的子代理会话 | `mode="session"` + `thread=true` |
| **会话恢复** | ACP的 `resumeSessionId` | ACP生成参数 |
| **Cron调度** | At/every/cron表达式支持 | `cron/types.ts` - `CronSchedule` |
| **会话清理器** | 自动清理会话过期 | `cron/session-reaper.ts` |
| **运行日志** | Cron执行历史 | `cron/run-log.ts` |

**关键特性**: **通过线程绑定的会话持久化** - 子代理可以创建绑定到消息线程的持久会话，实现长期运行的工作流。

### 空间可扩展性: ⭐⭐⭐⭐ (强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **子代理委托** | 带深度跟踪的 `sessions_spawn` 工具 | `subagent-spawn.ts` |
| **ACP委托** | Codex/Claude Code harness生成 | `acp-spawn.ts` |
| **深度限制** | `DEFAULT_SUBAGENT_MAX_SPAWN_DEPTH = 1` | `config/agent-limits.ts` |
| **最大子代理数** | `maxChildrenPerAgent = 5` 默认 | `subagent-spawn.ts` |
| **并行子代理** | 非阻塞生成带注册表跟踪 | `subagent-registry.ts` |
| **跨代理生成** | 用于代理目标的 `allowAgents` 配置 | `subagent-spawn.ts` |

**委托流程**:
```
父会话 (深度 0)
    ↓ sessions_spawn
子代理 (深度 1) - 默认MAX_DEPTH
    ✗ 不能进一步生成（被深度检查阻止）
    
或

父会话
    ↓ sessions_spawn runtime="acp"
ACP会话 (Codex/Claude Code)
    → 完整的代理harness，有自己的规划
```

### 交互可扩展性: ⭐⭐⭐⭐⭐ (优秀)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **多平台网关** | 15+消息平台 | `channels/`, 捆绑插件 |
| **会话绑定** | 线程/对话绑定 | `session-binding-service.ts` |
| **线程绑定会话** | 线程中的持久会话 | `subagent-spawn.ts` - `thread=true` |
| **跨平台交付** | Cron可以交付到任何平台 | `cron/delivery.ts` |
| **渠道适配器** | 基于插件的渠道支持 | `channels/plugins/` |
| **WebSocket SSE** | 实时会话更新 | 网关协议 |

**关键特性**: **平台无关的会话模型** - 会话从平台抽象出来，允许无缝迁移和跨平台连续性。

## 递归规划架构

### 当前实现

```
┌─────────────────────────────────────────────────────────────────┐
│                    父会话                                        │
│              (协调器 + 监控器混合)                               │
│         使用: sessions_spawn工具 → 注册表                        │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 生成
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                    子代理                                        │
│              (执行器 - 隔离上下文)                               │
│     深度 = 1, 阻止进一步委托                                     │
│     结果自动宣布给父代理                                         │
└──────────────────────────┬──────────────────────────────────────┘
                           │ 完成消息
                           ▼
              父注册表接收通知
              
或（对于复杂任务）:

┌─────────────────────────────────────────────────────────────────┐
│                    父会话                                        │
└──────────────────────────┬──────────────────────────────────────┘
                           │ sessions_spawn runtime="acp"
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│                    ACP会话                                       │
│           (完整Harness - Planner/Executor/Evaluator)             │
│              Codex/Claude Code及其自己的                          │
│              递归规划实现                                        │
└─────────────────────────────────────────────────────────────────┘
```

### 子代理生命周期钩子

| 钩子 | 时机 | 用途 |
|------|--------|---------|
| `subagent_spawning` | 生成前 | 线程绑定准备 |
| `subagent_spawned` | 生成后 | 通知/设置 |
| `subagent_ended` | 完成时 | 清理/告别 |

### 控制能力

**父代理可以**:
- 列出所有子代理（`subagents` 工具 - action=list）
- 终止子代理（`subagents` 工具 - action=kill）
- 引导子代理（`subagents` 工具 - action=steer）- 发送指导消息
- 接收完成消息
- 通过父流中继监控（用于ACP）

**子代理不能**:
- 生成更多子代理（深度限制）
- 访问父代理的会话历史
- 向任意渠道发送消息

## 自评估失真预防

| 风险因素 | 缓解措施 | 证据 |
|-------------|------------|----------|
| **自我评分** | 父代理监控子代理输出 | ACP的父流中继 |
| **无证据** | 子代理结果自动宣布 | `expectsCompletionMessage` 标志 |
| **上下文隔离** | 每个子代理的新鲜会话 | 带隔离上下文的 `childSessionKey` |
| **深度限制** | 硬限制防止无限递归 | `maxSpawnDepth` 检查 |
| **注册表跟踪** | 所有运行带元数据跟踪 | `subagent-registry.ts` |
| **工具限制** | 基于深度的能力系统 | `subagent-capabilities.ts` |

**与Claude Code / oh-my-codex / Hermes的比较**:

| 方面 | Claude Code | oh-my-codex | Hermes | OpenClaw |
|--------|-------------|-------------|--------|----------|
| Evaluator角色 | 独立验证代理 | 独立验证器 | 父代理双重角色 | 父代理 + 注册表监控 |
| 证据要求 | 命令输出 | 命令输出 | 摘要 | 自动宣布结果 |
| 状态隔离 | 进程/容器 | 工作树 + 角色 | 上下文 | 会话隔离 |
| 裁决格式 | `VERDICT: PASS/FAIL` | `VERDICT: PASS/FAIL/PARTIAL` | 文本摘要 | 结构化注册表条目 |
| 完成跟踪 | 验证代理 | 验证器角色 | 父代理判断 | 注册表 + 自动宣布 |

**风险评估**: **中低** - 基于注册表的跟踪和自动宣布系统提供了良好的可见性。然而，父代理同时充当协调器和隐式评估器，可能会造成一些失真风险。

## 解决的核心问题

### 1. 多平台代理部署 (交互可扩展性)

**问题**: 如何在许多消息平台上部署具有一致行为的代理。

**解决方案**:
- **网关抽象** - 所有平台通过统一网关注入
- **会话绑定服务** - 线程/对话绑定到会话
- **插件SDK** - 新平台的干净扩展点
- **Cron交付** - 计划任务可以到达任何平台

### 2. 大规模代理编排 (空间可扩展性)

**问题**: 如何协调多个代理处理相关任务。

**解决方案**:
- **子代理注册表** - 集中跟踪所有生成的代理
- **深度限制** - 防止失控递归
- **父流中继** - ACP子代理的实时监控
- **控制操作** - 用于运行时编排的列表/终止/引导
- **生命周期钩子** - 插件可扩展的生成/完成处理

### 3. 长期运行持久工作流 (时间可扩展性)

**问题**: 如何在 extended periods 内维护代理状态。

**解决方案**:
- **线程绑定会话** - 绑定到消息线程的持久会话
- **会话恢复** - ACP会话可以恢复
- **Cron调度** - 自然语言计划执行
- **会话清理器** - 自动清理防止资源泄漏

### 4. 安全委托边界 (控制机制)

**问题**: 如何委托工作而不失去控制或造成安全风险。

**解决方案**:
- **沙箱继承** - 沙箱父代理不能生成非沙箱子代理
- **能力衰减** - 基于深度限制工具
- **代理白名单** - 跨代理生成由配置控制
- **超时强制执行** - 每个子代理的运行超时

## 建议

### 值得效仿的优势

1. **以网关为中心的架构**: 网关（路由/交付）和代理（执行）之间的清晰分离是典范。

2. **子代理注册表**: 带元数据（任务、标签、超时、清理模式）的集中跟踪提供了出色的可观察性。

3. **多平台会话绑定**: 从平台抽象会话可实现强大的跨平台工作流。

4. **生命周期钩子**: 插件可扩展的生成/完成处理允许自定义编排逻辑。

5. **双运行时支持**: 支持内部子代理和外部ACP（Codex/Claude Code）harness提供了灵活性。

### 潜在改进

1. **独立Evaluator角色**: 考虑为关键任务添加明确的验证代理。

2. **Sprint合约**: 为子代理委托添加验收标准/合约系统。

3. **工作树隔离**: 考虑子代理文件操作的git工作树隔离。

4. **上下文压缩**: 虽然会话是隔离的，但长线程的显式上下文压缩可能有益。

5. **证据要求**: 结构化验证证据（测试输出、构建结果）vs. 文本摘要。

## 结论

OpenClaw代表了一个 **以交互可扩展性为中心的实现**，采用 **以网关为中心的Harness** 设计：

- ✅ **交互可扩展性**: 带会话绑定的最佳多平台网关
- ✅ **空间可扩展性**: 全面的子代理编排，带注册表和控制
- ✅ **集成**: 独特的能力生成外部ACP harnesses（Codex/Claude Code）
- ✅ **时间可扩展性**: 线程绑定持久会话，Cron调度
- ✅ **控制机制**: 多层次深度限制，沙箱继承，能力衰减
- ⚠️ **自评估失真**: 父代理充当隐式评估器，尽管注册表降低了风险

代码库展示了 **Claw Software的关注点**：在实际部署中跨 diverse messaging platforms 保持一致性，同时保持编排能力。双运行时支持（内部子代理 + 外部ACP）是一个独特的优势。

**Harness工程成熟度**: 在交互维度上为 **非常高**，在空间/时间维度上为 **高**。网关抽象和子代理注册表为多代理编排提供了坚实的基础。
