## 递归规划架构

### 三角色分离

```
┌─────────────────────────────────────────────────────────────┐
│                     PLANNER (Prometheus)                     │
│  - 将请求转化为可操作的工作计划                               │
│  - 写入 .omx/plans/*.md（其他人只读）                        │
│  - 不实现                                                    │
└──────────────────────────┬──────────────────────────────────┘
                           │ 带验收标准的计划
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                      EXECUTOR                                │
│  - 探索、实现、验证、完成                                    │
│  - "KEEP GOING UNTIL THE TASK IS FULLY RESOLVED"            │
│  - 必须提供验证证据                                          │
└──────────────────────────┬──────────────────────────────────┘
                           │ 实现 + 验证输出
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                      VERIFIER                                │
│  - 基于证据的验证                                            │
│  - 必需: 测试、类型检查、lint、构建证据                      │
│  - 裁决: PASS/FAIL/PARTIAL                                   │
└─────────────────────────────────────────────────────────────┘
```

### 角色约束（状态隔离）

| 角色 | 可以做 | 不能做 |
|------|--------|-----------|
| **Planner** | 写入计划到 `.omx/plans/`，询问权衡 | 编写代码、实现 |
| **Executor** | 实现、验证、运行测试 | 重写全局计划 |
| **Verifier** | 审查证据、发布裁决 | 修改实现 |

**Planner范围守卫**:
```markdown
<scope_guard>
- 仅写入计划到 `.omx/plans/*.md`
- 不编写代码文件
- 直到用户明确要求才生成最终计划
</scope_guard>
```

**Executor成功标准**:
```markdown
任务仅在以下情况下完成:
1. 实现了请求的行为
2. `lsp_diagnostics` 干净
3. 相关测试通过
4. 构建/类型检查成功
5. 没有临时/调试遗留物
6. **最终输出包括具体的验证证据**
```

## 自评估失真预防

| 风险因素 | 缓解措施 | 证据 |
|-------------|------------|----------|
| **自我评分** | 独立验证角色 | `verifier.ts` - 单独的验证步骤 |
| **合理化** | "无证据 = 未完成" 规则 | Executor提示: "Never claim success without tool-backed evidence" |
| **偏离目标** | 计划保存到 `.omx/plans/`（只读） | Planner约束: 执行期间计划是只读的 |
| **过早完成** | "KEEP GOING UNTIL FULLY RESOLVED" | Executor身份声明 |
| **过度投入偏见** | 最多3次修复尝试，然后升级 | `verifier.ts`: "如果3次尝试后仍然失败，则升级" |

**按任务大小的验证要求**:

| 大小 | 所需证据 |
|------|-------------------|
| **小** | 修改文件的类型检查，相关测试 |
| **标准** | 完整类型检查、测试套件、linter、端到端 |
| **大** | 完整类型检查、完整测试套件、安全审查、性能评估、API兼容性 |

## 解决的核心问题

### 1. 多代理协调 (空间可扩展性)

**问题**: 如何协调多个并行工作的代理而不造成混乱。

**解决方案**:
- **基于阶段的编排**: plan → prd → exec → verify → fix
- **角色专业化**: 不同阶段的不同代理
- **工作树隔离**: 每个工作器有隔离的git工作树
- **任务状态机**: 通过任务声明的独占所有权

### 2. 长期运行任务持久化 (时间可扩展性)

**问题**: 如何在数小时/数天内保持进度而不丢失状态。

**解决方案**:
- **Ralph持久化**: 带持久状态的所有者会话跟踪
- **`.omx/` 中的模式状态**: JSON状态文件在重启后存活
- **验证检查点**: 每次完成都需要证据
- **流水线阶段**: 有明确入口/出口标准的有界阶段

### 3. 意图澄清与执行分离

**问题**: 如何防止代理在理解之前实施。

**解决方案**:
- **$deep-interview**: 明确的澄清阶段
- **$ralplan**: 单独的计划，需要用户批准
- **Planner范围守卫**: 不能编写代码
- **询问闸门**: "只有在进展不可能时才询问"

### 4. 质量验证 (自评失真预防)

**问题**: 如何防止代理将不完整的工作合理化。

**解决方案**:
- **证据要求**: 没有命令输出 = 未验证
- **独立验证**: Verifier角色与executor分离
- **修复-验证循环**: 最多3次重试并升级
- **Lore提交**: 决策上下文保存在git trailers中

## 建议

### 值得效仿的优势

1. **明确的阶段编排**: `team-plan → team-prd → team-exec → team-verify → team-fix` 流水线是递归规划的清晰实现。

2. **基于角色的约束**: 在提示中使用 `<scope_guard>` 是一种优雅的方式，无需代码即可强制执行Harness边界。

3. **基于证据的验证**: "无证据 = 未完成" 规则有效地防止了自评估失真。

4. **持久状态管理**: `.omx/` 中的模式状态实现了真正的时间可扩展性。

5. **自主指令**: "自动进行而不询问" 最大化了吞吐量。

### 潜在改进

1. **明确的Sprint合约**: 虽然计划有验收标准，但它们可以像Claude Code的Sprint合约那样更明确地结构化。

2. **组件生命周期**: 功能标志或版本闸门，用于随着模型改进而退役角色（类似于Claude Code的功能标志）。

3. **收敛算法**: 记录合并并行工作的具体算法（cherry-pick, rebase, cross-rebase逻辑存在但可以更明确）。

## 结论

oh-my-codex代表了一个 **成熟的Harness优先实现**，专注于 **空间可扩展性**（空间可扩展性）:

- ✅ **空间可扩展性**: 阶段编排、角色专业化、工作树隔离
- ✅ **时间可扩展性**: Ralph持久化、模式状态、验证检查点
- ✅ **交互可扩展性**: AGENTS.md合约、角色范围守卫、自主指令
- ✅ **递归规划**: 清晰的Planner → Executor → Verifier分离
- ✅ **自评估失真预防**: 基于证据的验证协议

代码库展示了一种 **Cursor风格的方法** 来处理Harness工程 - 专注于并行代理执行的协调架构，具有强大的角色分离和基于阶段的编排。
