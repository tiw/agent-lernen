```
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                   框架层                                     │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐       │
│  │  工具系统    │  │  权限系统    │  │   上下文     │       │
│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘

│  └──────────────┘  └──────────────┘  └──────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 运行时层
- **模型API**: Anthropic SDK，支持流式传输、重试逻辑、令牌计数
- **执行**: Bun TypeScript运行时
- **UI**: React + Ink用于终端渲染

### 框架层
- **工具注册表**: 带有Zod模式的类型安全工具定义
- **权限系统**: 多模式权限处理（default/plan/bypassPermissions/auto/dontAsk）
- **上下文管理**: 文件状态缓存、对话历史

### Harness层
- **技能系统**: 可重用的工作流定义
- **代理系统**: 内置代理（general, verification, plan, explore）
- **协调器**: 具有分层委托的多代理编排

## 可扩展性维度评估

### 时间可扩展性: ⭐⭐⭐⭐⭐ (强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **上下文重置** | `compact/` 服务用于对话压缩 | 防止上下文窗口溢出 |
| **Sprint合约** | 验证代理的预定义标准 | 明确的 `VERDICT: PASS/FAIL` 输出 |
| **状态隔离** | Evaluator不与Generator共享状态 | `disallowedTools` 防止自我修改 |
| **独立评估** | 验证代理测试实际应用 | Playwright/MCP浏览器自动化 |
| **方向漂移保护** | 带任务通知的协调器模式 | 规划与执行的清晰分离 |

**三角色实现**:
```
协调器 (Planner) → 工作代理 (Generator) → 验证代理 (Evaluator)
```

验证代理是 **纯Evaluator**:
- 不能修改文件 (`disallowedTools: [FILE_EDIT, FILE_WRITE, ...]`)
- 必须测试实际应用行为
- 以客观的 `VERDICT: PASS/FAIL/PARTIAL` 结束

### 空间可扩展性: ⭐⭐⭐⭐ (强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **递归分解** | 协调器模式生成子规划器 | 带有 `description` + `prompt` 的 `AgentTool` |
| **并行执行** | 后台模式的异步代理启动 | `run_in_background: true` |
| **工作树隔离** | 每个代理一个Git工作树 | `utils/worktree.ts` 中的 `createAgentWorktree()` |
| **收敛** | 带结构化结果的任务通知 | 基于XML的 `<task-notification>` 格式 |
| **多代理团队** | TeamCreate/TeamDelete工具 | `TeamCreateTool`, `TeamDeleteTool` |

**架构模式**:
```typescript
// 协调器生成并行工作器
AgentTool({ description: "研究认证", prompt: "..." })
AgentTool({ description: "研究存储", prompt: "..." })

// 工作器在完成后通知
<task-notification>
  <task-id>agent-a1b</task-id>
  <status>completed</status>
  <result>...</result>
</task-notification>
```

### 交互可扩展性: ⭐⭐⭐⭐⭐ (强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **环境治理** | 权限规则作为可执行约束 | `Bash(rm:*)`, `Bash(npm publish:*)` 模式 |
| **知识版本控制** | AGENTS.md，仓库中的技能 | 基于Markdown的代理定义 |
| **基于约束的控制** | 权限模式 + 规则 | `default`/`plan`/`bypassPermissions`/`auto`/`dontAsk` |
| **吞吐量优先** | 带分类器的自动模式 | AI分类器用于权限决策 |

**约束示例**:
```typescript
// 比"小心使用rm"更有效
{ toolName: "Bash", ruleContent: "rm:*", behavior: "ask" }

// 比"未经批准不要发布"更有效
{ toolName: "Bash", ruleContent: "npm publish:*", behavior: "ask" }
```

**四个基础共识**:

| 原则 | 实现 | 状态 |
|-----------|----------------|--------|
| **杠杆点转移** | 人类设计AGENTS.md，不编写代码 | ✅ 完整 |
| **知识版本控制** | 代理/技能在仓库Markdown中定义 | ✅ 完整 |
| **约束优于指令** | 带可执行模式的权限规则 | ✅ 完整 |
| **吞吐量优于完美** | 自动模式分类器接受计算风险 | ✅ 完整 |

## 递归规划架构

### 角色分离

| 角色 | 组件 | 职责 | 隔离 |
|------|-----------|------------------|-----------|
| **Planner** | 协调器模式 | 分解任务、生成工作器、综合结果 | 不直接生成代码 |
| **Generator** | 工作代理 | 根据提示实现 | 没有评估权限 |
| **Evaluator** | 验证代理 | 根据Sprint合约测试 | 不与Generator共享状态 |

### 验证代理架构

```typescript
// 纯EVALUATOR - 不能修改项目
export const VERIFICATION_AGENT: BuiltInAgentDefinition = {
  agentType: 'verification',
  disallowedTools: [
    AGENT_TOOL_NAME,      // 不能生成子代理
    FILE_EDIT_TOOL_NAME,  // 不能修改文件
    FILE_WRITE_TOOL_NAME, // 不能创建文件
    // ... 其他写入工具
  ],
  getSystemPrompt: () => VERIFICATION_SYSTEM_PROMPT,
  // 必须以 VERDICT: PASS/FAIL/PARTIAL 结束
}
```

**状态隔离机制**:
1. **工具限制**: 明确的 `disallowedTools` 列表
2. **进程隔离**: 独立的代理上下文
3. **基于合约的评估**: 预定义的验收标准
4. **真实应用测试**: Playwright/MCP浏览器自动化

## 自评估失真预防

| 风险因素 | 缓解措施 | 证据 |
|-------------|------------|----------|
| **自我评分** | 独立验证代理 | 验证代理不能是实现者 |
| **合理化** | 明确的命令输出要求 | "没有Command运行块的检查不是PASS" |
| **偏离目标** | 预定义的Sprint合约 | 工作开始前设定标准 |
| **过度投入偏见** | 新鲜的代理进行验证 | 与实现者没有共享上下文 |

**验证提示摘录**:
```
=== 识别你自己的合理化 ===
你会感到想要跳过检查的冲动。这些正是你寻找的借口：
- "根据我的阅读，代码看起来正确" —— 阅读不是验证。运行它。
- "实现者的测试已经通过" —— 实现者是LLM。独立验证。
- "这可能没问题" —— 可能不等于已验证。运行它。
```

## 解决的核心问题

### 1. 长期运行代理一致性 (时间可扩展性)

**问题**: 代理在数小时工作后偏离目标。

**解决方案**: 
- 上下文压缩服务 (`compact/`)
- 带验证检查点的Sprint执行
- 独立Evaluator角色防止自评估失真

### 2. 多代理协调 (空间可扩展性)

**问题**: 数百个代理同时工作造成混乱。

**解决方案**:
- 具有分层委托的协调器模式
- 并行开发的工作树隔离
- 用于收敛的结构化任务通知
- 代理组的团队管理工具

### 3. 大规模权限 (交互可扩展性)

**问题**: 人类无法审查来自数百个代理的每个工具调用。

**解决方案**:
- 可执行权限规则（约束 > 指令）
- AI分类器用于自动模式权限决策
- 带手动审批回退的拒绝跟踪
- 针对不同风险容忍度的多种权限模式

### 4. 自评估失真 (自评失真)

**问题**: 代理发现缺陷但将其合理化为可接受。

**解决方案**:
- 具有明确工具限制的验证代理
- 强制命令执行进行验证
- 预定义的Sprint合约
- Generator和Evaluator之间的状态隔离

## 建议

### 值得效仿的优势

1. **验证代理模式**: 带有明确disallowedTools的三角色分离是递归规划的典范实现。

2. **权限规则系统**: 基于约束的方法（`Bash(rm:*)`）比自然语言指令更具可扩展性。

3. **功能标志生命周期**: 使用 `bun:bundle` 功能标志可以在模型改进时优雅地退役组件。

4. **结构化通信**: 基于XML的任务通知提供了清晰的收敛机制。

### 潜在改进

1. **明确的Sprint合约**: 虽然验证有预定义标准，但主代理循环可以从更明确的Sprint合约定义中受益。

2. **组件生命周期文档**: 功能标志存在，但可以更明确地映射到模型能力演进。

3. **收敛算法**: 记录合并并行代理输出的具体算法（目前在协调器模式中是隐式的）。

## 结论

Claude Code代表了一个 **成熟的Harness优先实现**，与Harness工程统一框架高度一致：

- ✅ **时间可扩展性**: 独立验证、上下文管理、sprint合约
- ✅ **空间可扩展性**: 协调器模式、工作树隔离、团队管理
- ✅ **交互可扩展性**: 可执行约束、权限规则、自动模式
- ✅ **递归规划**: 完整的Planner → Generator → Evaluator分离
- ✅ **自评估失真预防**: 状态隔离的验证代理

代码库展示了Anthropic对 **运行时修正**（时间可扩展性）作为主要维度的关注，并辅以空间和交互可扩展性的实现。
