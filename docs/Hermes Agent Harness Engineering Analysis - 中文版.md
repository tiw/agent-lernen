## 执行摘要

| 方面 | 评估 |
|--------|------------|
| **主要维度** | 时间可扩展性 - 上下文管理和持久化 |
| **次要维度** | 空间维度（委托）+ 交互维度（多平台） |
| **递归规划** | **部分实现** - 子代理委托但没有明确的Evaluator |

## 控制论映射

### 1. 反馈系统

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **上下文压缩** | `agent/context_compressor.py` | 对话历史的自动摘要 |
| **记忆管理器** | `agent/memory_manager.py` | 多提供者记忆编排与召回 |
| **会话搜索** | `hermes_state.py` | 跨过去对话的FTS5 SQLite搜索 |

**关键洞察**: 上下文压缩器实现了 **迭代摘要** - 在多次压缩中保留信息，这是长期运行会话的复杂反馈循环。

### 2. 控制机制

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **迭代预算** | `run_agent.py` - `IterationBudget` 类 | 线程安全的迭代计数器（父代理默认90，子代理50） |
| **委托深度限制** | `tools/delegate_tool.py` | `MAX_DEPTH = 2` - 父→子→孙被拒绝 |
| **子代理阻止的工具** | `tools/delegate_tool.py` | `DELEGATE_BLOCKED_TOOLS` - 防止递归委托、记忆写入 |
| **检查点快照** | `tools/checkpoint_manager.py` | 影子git仓库用于透明回滚 |
| **审批系统** | `tools/approval.py` | 危险命令检测与用户确认 |

**关键洞察**: 委托系统实现了 **受约束的自主性** - 子代理获得受限工具集且不能递归委托。

### 3. 通信基础设施

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **多平台网关** | `gateway/` 目录 | Telegram, Discord, Slack, WhatsApp, Signal适配器 |
| **MCP集成** | `tools/mcp_tool.py` | 模型上下文协议用于外部工具服务器 |
| **AGENTS.md合约** | `AGENTS.md` | AI助手的顶级开发指南 |
| **技能Markdown** | `skills/` 目录 | YAML frontmatter + markdown用于技能定义 |
| **Cron交付** | `cron/scheduler.py` | 计划任务交付到任何平台 |

**关键洞察**: 多平台支持展示了 **交互可扩展性** - 同一个代理实例同时处理多个平台的对话。

### 4. 熵管理

| 实现 | 位置 | 描述 |
|----------------|----------|-------------|
| **上下文修剪** | `agent/context_compressor.py` | LLM摘要前的工具输出修剪（廉价预传递） |

## 可扩展性维度评估

### 时间可扩展性: ⭐⭐⭐⭐⭐ (非常强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **上下文压缩** | 带迭代更新的结构化摘要 | 带有 `_previous_summary` 的 `ContextCompressor` 类 |
| **令牌预算保护** | 按令牌数而非消息数的尾部保护 | `tail_token_budget` 与上下文成比例 |
| **模型感知限制** | 每模型上下文长度跟踪 | `agent/model_metadata.py` - 上下文长度查询 |
| **会话持久化** | 带FTS5搜索的SQLite SessionDB | `hermes_state.py` - 完整对话历史 |
| **检查点回滚** | 用于文件系统快照的影子git仓库 | `tools/checkpoint_manager.py` |
| **记忆提供者系统** | 可插拔记忆（内置+外部） | `agent/memory_manager.py` |

**上下文压缩算法**:
```python
# 1. 修剪旧的工具结果（廉价，无需LLM调用）
# 2. 保护头部消息（系统+第一次交换）
# 3. 按令牌预算保护尾部（最近约20K令牌）
# 4. 用结构化LLM提示摘要中间轮次
# 5. 在后续压缩中迭代更新之前的摘要
```

**结构化摘要模板**:
- 目标
- 进展
- 决策
- 文件
- 下一步

### 空间可扩展性: ⭐⭐⭐ (中等)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **子代理委托** | 带隔离上下文的 `delegate_task` 工具 | `tools/delegate_tool.py` |
| **并行执行** | 批量委托的 `ThreadPoolExecutor` | delegate_tool中的 `concurrent.futures` |
| **深度限制** | `MAX_DEPTH = 2` 防止无限递归 | delegate_tool中的硬限制 |
| **工具限制** | 子代理获得阻止的工具集 | `DELEGATE_BLOCKED_TOOLS` frozenset |
| **批量模式** | 并行多个子代理 | 委托中的 `batch` 参数 |

**局限性**:
- 没有明确的 **Planner → Executor → Evaluator** 分离
- 父代理充当隐式协调器但也进行验证
- 没有工作树隔离（依赖文件系统状态）
- 子代理共享父代理的终端环境

### 交互可扩展性: ⭐⭐⭐⭐⭐ (非常强)

| 能力 | 实现 | 证据 |
|------------|----------------|----------|
| **多平台网关** | Telegram, Discord, Slack, WhatsApp, Signal等 | `gateway/platforms/` 目录 |
| **跨平台连续性** | 会话搜索跨平台工作 | `hermes_state.py` |
| **Cron调度** | 自然语言计划任务 | `cron/scheduler.py` |
| **语音备忘录支持** | 转录集成 | README中提到 |
| **平台特定技能** | 平台条件技能加载 | skill_utils中的 `skill_matches_platform()` |

## 递归规划架构

### 当前实现

```
┌─────────────────────────────────────────────────────────────┐
│                    父代理                                    │
│              (协调器 + Evaluator混合)                        │
│                     IterationBudget(90)                     │
└──────────────────────────┬──────────────────────────────────┘
                           │ delegate_task
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                     子代理                                   │
│              (执行器 - 隔离上下文)                           │
│         IterationBudget(50) + 阻止的工具                    │
│     不能: 委托、写入记忆、发送消息                          │
└──────────────────────────┬──────────────────────────────────┘
                           │ 结果摘要
                           ▼
                    父代理评估
```

### 委托约束

**子代理阻止的工具** (`DELEGATE_BLOCKED_TOOLS`):
- `delegate_task` - 无递归委托
- `clarify` - 无用户交互
- `memory` - 不写入共享MEMORY.md
- `send_message` - 无跨平台副作用
- `execute_code` - 应逐步推理

**这实现了什么**:
- 防止无限递归
- 将用户交互集中在父代理
- 保持记忆一致性
- 控制子代理操作的影响范围

**缺少什么**:
- 没有独立的Evaluator角色
- 父代理同时进行协调和验证
- 没有明确的Sprint合约
- 验证依赖父代理的判断

## 自评估失真预防

| 风险因素 | 缓解措施 | 证据 |
|-------------|------------|----------|
| **自我评分** | 父代理审查子代理摘要 | 无独立Evaluator |
| **无证据要求** | 子代理提供摘要而非命令输出 | 摘要是描述性的，非基于证据 |
| **上下文隔离** | 子代理有隔离的对话历史 | 新鲜上下文，无父代理历史 |
| **深度限制** | `MAX_DEPTH = 2` 防止过多委托 | 在孙代理处硬停止 |
| **迭代预算** | 父代理(90)与子代理(50)的单独预算 | `IterationBudget` 类 |

**与Claude Code / oh-my-codex的比较**:

| 方面 | Claude Code | oh-my-codex | Hermes |
|--------|-------------|-------------|--------|
| Evaluator角色 | 独立验证代理 | 独立验证器角色 | 父代理（双重角色） |
| 证据要求 | 命令输出 | 命令输出 | 摘要描述 |
| 状态隔离 | 进程/容器 | 工作树 + 角色守卫 | 仅上下文隔离 |
| 裁决格式 | `VERDICT: PASS/FAIL` | `VERDICT: PASS/FAIL/PARTIAL` | 文本摘要 |

**风险评估**: **中等** - 父代理同时充当协调器和评估器，可能导致自评估失真。没有基于证据要求的独立验证。

## 解决的核心问题

### 1. 长期运行会话管理 (时间可扩展性)

**问题**: 如何在数小时/数天内保持连贯对话而不溢出上下文。

**解决方案**:
- **上下文压缩** 带结构化摘要
- **迭代摘要更新**（在压缩中保留信息）
- **令牌预算尾部保护**（不仅是消息计数）
- **模型感知限制**（不同模型有不同的上下文长度）
- **带FTS5的SessionDB**（搜索过去的对话）

### 2. 多平台存在 (交互可扩展性)

**问题**: 如何在许多消息平台上提供一致的代理体验。

**解决方案**:
- **网关架构** 带平台适配器
- **统一会话状态** 在SQLite中（跨平台工作）
- **Cron调度** 带平台交付
- **平台特定技能加载**

### 3. 安全子代理执行 (空间可扩展性 - 部分)

**问题**: 如何委托工作而不失去控制。

**解决方案**:
- **工具阻止** - 子代理不能递归委托
- **上下文隔离** - 每个子代理的新鲜对话
- **迭代预算** - 父/子的单独限制
- **深度限制** - 2层的硬停止

**差距**: 没有明确的Planner → Executor → Evaluator分离。父代理同时进行协调和验证。

### 4. 持久记忆 (熵管理)

**问题**: 如何跨会话积累知识。

**解决方案**:
- **技能系统** - 从经验中获得的程序性记忆
- **记忆提供者** - 可插拔语义记忆
- **会话搜索** - FTS5全文搜索
- **检查点快照** - 文件系统回滚能力

## 建议

### 值得效仿的优势

1. **上下文压缩算法**: 带结构化模板（目标、进展、决策、文件、下一步）的迭代摘要是典范。

2. **模型感知上下文管理**: 跟踪每个模型/提供者的上下文长度可实现最优令牌使用。

3. **多平台网关**: Telegram/Discord/Slack等的干净抽象是交互可扩展性的很好示例。

4. **检查点管理器**: 影子git仓库提供透明回滚而不污染用户目录。

5. **可插拔记忆提供者**: 带内置+外部提供者支持的 `MemoryManager` 显示了良好的可扩展性设计。

### 潜在改进

1. **独立Evaluator角色**: 添加类似于Claude Code验证代理或oh-my-codex验证器的验证角色。

2. **基于证据的验证**: 需要命令输出证据，而不仅仅是描述性摘要。

3. **Sprint合约**: 在子代理委托之前定义明确的验收标准。

4. **工作树隔离**: 考虑子代理的git工作树隔离（如oh-my-codex）。

5. **明确的三角色分离**: 将Planner、Executor和Evaluator分离为不同的角色。

## 比较总结

| 维度 | Claude Code | oh-my-codex | Hermes Agent |
|-----------|-------------|-------------|--------------|
| **主要焦点** | 时间维度（验证） | 空间维度（协调） | 时间维度（上下文管理） |
| **递归规划** | 完整3角色分离 | 完整3角色分离 | 部分（无Evaluator） |
| **上下文管理** | 压缩+验证 | 模式状态+检查点 | **最佳: 迭代摘要** |
| **多平台** | 仅CLI | 仅CLI | **最佳: 多平台网关** |
| **验证** | 独立代理 | 独立验证器 | 父代理双重角色 |
| **计划任务** | Cron支持 | Cron支持 | **最佳: 内置自然语言cron** |

---

## 结论

Hermes Agent代表了一个 **以时间可扩展性为中心的实现**，采用 **模型感知Harness** 设计：

- ✅ **时间可扩展性**: 带迭代摘要的最佳上下文压缩
- ✅ **交互可扩展性**: 全面的多平台网关
- ✅ **记忆管理**: 可插拔语义+程序性记忆
- ⚠️ **空间可扩展性**: 基本委托，无完整角色分离
- ⚠️ **自评估失真**: 父代理双重角色造成中等风险

代码库展示了 **Nous Research的关注点**：长期运行、持久的代理，可以在 extended periods 内跨多个平台工作。上下文压缩系统特别复杂，具有模型感知限制和迭代摘要更新。

**Harness工程成熟度**: 在时间/交互维度上为 **高**，在空间维度上为 **中等**，因为缺乏独立的Evaluator角色。
