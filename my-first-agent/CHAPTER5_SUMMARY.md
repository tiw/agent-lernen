## 🧠 核心概念

### 1. 缓存命中是架构原则，不是优化

| 设计 | 缓存考虑 | 实现状态 |
|------|----------|----------|
| 状态冻结（pinCacheEdits） | 确保 Prompt 前缀字节一致 | ✅ 已实现 |
| 缓存编辑压缩（cache_edits） | 服务器端删除，本地不变 | ✅ 已实现 |
| 14 个缓存断点 | 精确控制缓存边界 | ✅ 已实现 |
| 粘性锁存器 | 防止模式切换导致失效 | ✅ 已实现 |
| 只运行主线 | 避免子代理破坏主线程缓存 | ✅ 已实现 |

**为什么缓存这么重要？** Anthropic API 的 Prompt Caching 可以节省 **50-90% 的 API 成本**。

### 2. Token 估算策略

| 方法 | 精度 | 速度 | 成本 | 适用场景 | 实现状态 |
|------|------|------|------|----------|----------|
| 粗糙估算 | ~80% | 极快 | 零 | 实时监控 | ✅ 4 chars/token |
| 类型感知 | ~90% | 快 | 零 | JSON 等特殊格式 | ✅ JSON 2 chars/token |
| API 精确计数 | 100% | 慢 | 免费 | 压缩决策 | ✅ Anthropic API |

**关键洞察**：JSON 的 token 密度是普通文本的 2 倍（因为大量使用标点符号）。

### 3. 工具结果预算

```
工具执行 → 结果超过阈值？
  ├─ 是 → 写磁盘 + 返回 2KB 预览
  └─ 否 → 直接返回完整结果
```

**状态冻结**：一旦决定用预览，后续所有 API 调用都用同样的预览文本，确保 Prompt 前缀字节一致。

## 💻 已实现代码

### 1. TokenEstimator（Token 估算器）

```python
class TokenEstimator:
    """Token 估算器"""

    DEFAULT_BYTES_PER_TOKEN = 4
    DENSE_BYTES_PER_TOKEN = 2  # JSON

    def estimate(self, text: str) -> int:
        """粗糙估算：tokens ≈ len(text) / 4"""
        return math.ceil(len(text) / self.DEFAULT_BYTES_PER_TOKEN)

    def estimate_for_type(self, text: str, file_type: str = '') -> int:
        """类型感知的 token 估算"""
        ext = file_type.lower().lstrip('.')
        if ext in ('json', 'jsonl', 'jsonc'):
            return math.ceil(len(text) / self.DENSE_BYTES_PER_TOKEN)
        return self.estimate(text)
```

## 💡 学习心得

### 核心收获

1. **七层架构设计精妙** — 每一层都是上一层的"安全网"
2. **缓存命中是经济命脉** — 节省 50-90% API 成本
3. **状态冻结是关键设计** — 确保 Prompt 前缀字节一致
4. **Token 估算需要分层** — 日常估算用粗糙，压缩决策用精确
5. **零 API 压缩策略** — 会话记忆实时结构化，压缩时直接用

### 设计启发

1. **预防优于治疗** — 第 1 层不让大结果进上下文
2. **渐进式压缩** — 从微压缩到全压缩，逐步升级
3. **缓存断点精确控制** — 14 个断点，最大化缓存命中
4. **粘性锁存器防抖动** — 防止模式切换导致缓存失效

### 难点攻克

1. **理解缓存编辑压缩** — 服务器端删除 vs 本地不变
2. **粘性锁存器机制** — 如何平衡缓存命中与内容更新
3. **触发条件设计** — Token 增长 + 工具调用的组合判断
